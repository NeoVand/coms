# Graph Query Language (GraphQL): A Deep, Citation-Backed Brief for Smart Engineers

## Prerequisites and glossary

Before GraphQL clicks, an engineer should be comfortable with the layer cake on which it sits and the vocabulary it borrows from type theory and database querying.

- **OSI / TCP-IP layers** — GraphQL is overwhelmingly an *application-layer* (L7) protocol that runs on top of a *transport-layer* (L4) byte stream, almost always TCP, almost always wrapped in TLS. The GraphQL spec deliberately *does not* mandate a transport ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/)). The de-facto transport binding is GraphQL-over-HTTP ([https://graphql.github.io/graphql-over-http/draft/](https://graphql.github.io/graphql-over-http/draft/)). Authoritative refresher: RFC 9293 (TCP) and the WHATWG Fetch spec. [GraphQL](https://graphql.org/learn/response/)[Graphql](https://graphql.github.io/graphql-over-http/draft/)
- **HTTP/1.1, HTTP/2, HTTP/3** — Request/response protocol that gives us methods, headers, status codes, and (in /2 and /3) multiplexing. GraphQL typically uses POST `/graphql` (and optionally GET for cacheable, idempotent queries) ([https://graphql.org/learn/serving-over-http/](https://graphql.org/learn/serving-over-http/)). [GraphQL](https://graphql.org/learn/serving-over-http/)
- **TLS** — Transport security on port 443. GraphQL inherits TLS from its host, the same way REST does.
- **WebSocket (RFC 6455)** — Bidirectional framed stream over a single TCP connection used to carry GraphQL subscriptions via the `graphql-ws` sub-protocol ([https://github.com/enisdenjo/graphql-ws](https://github.com/enisdenjo/graphql-ws)).
- **Server-Sent Events (SSE)** — One-way HTTP stream of `text/event-stream` events. Increasingly the preferred subscription transport because it survives proxies that mangle WebSocket upgrades ([https://gqlgen.com/recipes/subscriptions/](https://gqlgen.com/recipes/subscriptions/), [https://the-guild.dev/graphql/yoga-server/docs/features/defer-stream](https://the-guild.dev/graphql/yoga-server/docs/features/defer-stream)). [Gqlgen](https://gqlgen.com/recipes/subscriptions/)
- **JSON (RFC 8259)** — The default serialization for GraphQL request bodies and responses ([https://graphql.org/learn/response/](https://graphql.org/learn/response/)). [GraphQL](https://graphql.org/learn/response/)
- **Socket / handshake / frame / stream** — Generic networking terms; for GraphQL specifically, "frame" matters when discussing WebSocket subscription messages, and "stream" is the verb of choice for `@stream`/incremental delivery ([https://github.com/graphql/graphql-spec/pull/1110](https://github.com/graphql/graphql-spec/pull/1110)).
- **Schema** — A strongly-typed description of every type and field a server exposes ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/)).
- **SDL (Schema Definition Language)** — The textual notation for GraphQL schemas; became part of the spec in February 2018 ([https://en.wikipedia.org/wiki/GraphQL](https://en.wikipedia.org/wiki/GraphQL)). [Wikipedia](https://en.wikipedia.org/wiki/GraphQL)
- **Type system primitives** — *scalars* (Int, Float, String, Boolean, ID), *objects*, *interfaces*, *unions*, *enums*, *input objects*, *lists* (`[T]`), *non-null wrappers* (`T!`) ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/)).
- **Query / Mutation / Subscription** — The three operation types: read, write, and long-lived event stream respectively ([https://en.wikipedia.org/wiki/GraphQL](https://en.wikipedia.org/wiki/GraphQL)).
- **Resolver** — Server-side function that produces the value for a single field; depth-first execution traverses the operation tree calling each resolver ([https://www.graphql-js.org/docs/n1-dataloader/](https://www.graphql-js.org/docs/n1-dataloader/)).
- **Introspection** — A standardized meta-query (`__schema`, `__type`) that returns the schema itself; powers GraphiQL and codegen ([https://docs.github.com/en/graphql/overview/about-the-graphql-api](https://docs.github.com/en/graphql/overview/about-the-graphql-api)).
- **Persisted query / Automatic Persisted Query (APQ)** — Send a SHA-256 hash instead of the full query string. APQ is a *performance* feature; explicit safelisted persisted queries are a *security* feature ([https://www.apollographql.com/docs/apollo-server/performance/apq](https://www.apollographql.com/docs/apollo-server/performance/apq), [https://www.apollographql.com/docs/graphos/operations/persisted-queries](https://www.apollographql.com/docs/graphos/operations/persisted-queries)). [Apollo GraphQL](https://www.apollographql.com/docs/react/api/link/persisted-queries)
- **Federation / supergraph / subgraph** — Architecture for composing many small GraphQL services into one client-facing graph ([https://www.apollographql.com/blog/apollo-federation-2-is-now-generally-available](https://www.apollographql.com/blog/apollo-federation-2-is-now-generally-available)).
- **Schema Coordinates / OneOf input objects** — Two formal additions in the September 2025 edition of the spec ([https://progosling.com/en/dev-digest/graphql-spec-september-2025](https://progosling.com/en/dev-digest/graphql-spec-september-2025)).

## History and story

**1. The Facebook years (2012–2015).** In the spring of 2012, three engineers at Facebook — **Nick Schrock, Lee Byron, and Dan Schafer** (note: the often-misstated "Dan Schmidt" is actually Dan Schafer) — set out to build a new data-fetching API for the iOS News Feed rewrite, because the old HTML5 wrapper apps were collapsing under their own weight ([https://medium.com/@leeb/introducing-the-graphql-foundation-3235d8186d6d](https://medium.com/@leeb/introducing-the-graphql-foundation-3235d8186d6d), [https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/](https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/)). Schrock's first prototype, internally called **"SuperGraph"**, was posted for Facebook code review in February 2012 ([https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/](https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/)). GraphQL 1.0 shipped to production with Facebook for iOS 5.0 in summer 2012 ([https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/](https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/)). By the end of 2014 every component of Facebook's iOS app was served by GraphQL, and Lee Byron later joked that the original team "essentially became API designers" as each new product team came asking for help ([https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/](https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/)). [Medium + 5](https://medium.com/@leeb/introducing-the-graphql-foundation-3235d8186d6d)

**2. Public release (2015).** GraphQL was first publicly described at **React.js Conf, Facebook HQ, January 28–29, 2015**, in the now-famous Daniel Schafer / Jing Chen talk, alongside the unveiling of Relay and React Native ([https://legacy.reactjs.org/blog/2015/02/18/react-conf-roundup-2015.html](https://legacy.reactjs.org/blog/2015/02/18/react-conf-roundup-2015.html)). A revised draft spec and the **GraphQL.js reference implementation** were published in July 2015, and Lee Byron's official "GraphQL: A data query language" announcement landed September 14, 2015 ([https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/](https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/)). [React](https://legacy.reactjs.org/blog/2015/02/18/react-conf-roundup-2015.html)

**3. The license fight (2017).** Facebook initially shipped GraphQL under its controversial **BSD + Patents** license, which the Apache Software Foundation banned from Apache projects in July 2017 and which led GitLab to **freeze its GraphQL effort** ([https://nordicapis.com/what-the-graphql-patent-release-means-for-the-api-industry/](https://nordicapis.com/what-the-graphql-patent-release-means-for-the-api-industry/), [https://www.theregister.com/2017/09/22/facebook_will_free_react_other_code_from_unloved_license/](https://www.theregister.com/2017/09/22/facebook_will_free_react_other_code_from_unloved_license/)). On **September 26, 2017**, Facebook relicensed the GraphQL specification under the **Open Web Foundation Agreement (OWFa) v1.0** and relicensed graphql-js and Relay under **MIT** ([https://leebyron.com/relicensing-the-graphql-specification/](https://leebyron.com/relicensing-the-graphql-specification/), [https://www.apollographql.com/blog/facebook-grants-full-patent-rights-to-all-graphql-users-2c41abc6df66](https://www.apollographql.com/blog/facebook-grants-full-patent-rights-to-all-graphql-users-2c41abc6df66), [https://www.theregister.com/2017/09/26/facebook_license_surgery_on_react/](https://www.theregister.com/2017/09/26/facebook_license_surgery_on_react/)). ## How it actually works

GraphQL is three things at once: **a query language**, **an execution engine**, and **a type system** — and the spec is explicit that it is *not* tied to HTTP, JSON, or any database ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/)). [Graphql](https://spec.graphql.org/September2025/)

**Document structure.** A GraphQL *document* contains one or more *operations* (`query`, `mutation`, `subscription`) and zero or more *fragments*. Each operation has a *selection set* of *fields*; each field can take *arguments*, can be aliased, and can carry *directives* (`@include`, `@skip`, `@deprecated`, `@oneOf`, plus user-defined ones). *Variables* (`$id: ID!`) parameterize operations ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/), [https://relay.dev/graphql/connections.htm](https://relay.dev/graphql/connections.htm)). [Graphql](https://spec.graphql.org/September2025/)

**SDL.** Schemas are written as:

```
type Query { user(id: ID!): User }
type User { id: ID! name: String! posts: [Post!]! }
```

Non-null is `!`; lists are `[T]`. Trailing commas are *valid in argument and field lists* — a small quirk that helps codegen ([https://en.wikipedia.org/wiki/GraphQL](https://en.wikipedia.org/wiki/GraphQL)). [Wikipedia](https://en.wikipedia.org/wiki/GraphQL)

**Type system primitives.** Scalars (Int, Float, String, Boolean, ID, plus custom), objects, interfaces, unions, enums, input objects, lists, non-null wrappers ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/)). [Wikipedia](https://en.wikipedia.org/wiki/GraphQL)

**Validation phase.** Before execution, the server parses the document, then validates it against the schema (every field exists, every argument is type-correct, no overlapping incompatible selections, fragments are well-formed). Validation is where Schema Coordinates (Sept 2025) make tooling meaningful ([https://progosling.com/en/dev-digest/graphql-spec-september-2025](https://progosling.com/en/dev-digest/graphql-spec-september-2025)). [Progosling](https://progosling.com/en/dev-digest/graphql-spec-september-2025)

**Execution phase.** The server walks the validated operation depth-first, calling a *resolver* for every requested field. Field-level errors are caught and pushed onto a top-level `errors[]` array — the response can carry **partial data alongside errors** ([https://graphql.org/learn/response/](https://graphql.org/learn/response/)). The canonical envelope is `{ "data": ..., "errors": [...], "extensions": {...} }`. [GraphQL](https://graphql.org/learn/response/)

**Wire format — the GraphQL-over-HTTP spec.** The draft (working draft Jul 24, 2025) introduces the new media type **`application/graphql-response+json`**, recommends `/graphql` as the URL, allows GET (for queries only, with parameters URL-encoded; `variables` and `extensions` as JSON strings) and POST, and sets clear status-code rules ([https://graphql.github.io/graphql-over-http/draft/](https://graphql.github.io/graphql-over-http/draft/), [https://progosling.com/en/dev-digest/graphql-over-http-working-draft-2025](https://progosling.com/en/dev-digest/graphql-over-http-working-draft-2025)). Key rule: when responding with the new media type, **a non-null `data` MUST yield 200** (200 plus errors is a *successful partial response*); 4xx/5xx are reserved for failures that prevented producing a well-formed GraphQL response ([https://github.com/graphql/graphql-over-http/blob/main/spec/GraphQLOverHTTP.md](https://github.com/graphql/graphql-over-http/blob/main/spec/GraphQLOverHTTP.md)). Legacy `application/json` is still accepted, but from January 1, 2025 servers SHOULD support the new media type and treat unspecified `Accept` headers as if they had requested it ([https://github.com/graphql/graphql-over-http/blob/a1e6d8ca248c9a19eb59a2eedd988c204909ee3f/spec/GraphQLOverHTTP.md](https://github.com/graphql/graphql-over-http/blob/a1e6d8ca248c9a19eb59a2eedd988c204909ee3f/spec/GraphQLOverHTTP.md)). [Progosling + 4](https://progosling.com/en/dev-digest/graphql-over-http-working-draft-2025)

**Subscriptions transports.**

- **`subscriptions-transport-ws`** — original Apollo protocol, **deprecated** and unmaintained ([https://www.apollographql.com/docs/react/data/subscriptions](https://www.apollographql.com/docs/react/data/subscriptions), [https://github.com/enisdenjo/graphql-ws](https://github.com/enisdenjo/graphql-ws)). [GitHub](https://github.com/nestjs/graphql/issues/2775)
- **`graphql-ws`** — the modern, "GraphQL over WebSocket Protocol"-compliant library ([https://github.com/enisdenjo/graphql-ws](https://github.com/enisdenjo/graphql-ws), [https://www.npmjs.com/package/graphql-ws](https://www.npmjs.com/package/graphql-ws)). Cross-incompatible with the old one. [npm](https://www.npmjs.com/package/graphql-ws)[npm](https://www.npmjs.com/package/graphql-ws)
- **SSE (`text/event-stream`)** — encouraged by the GraphQL-over-HTTP working group and shipping in Yoga, gqlgen, Spring for GraphQL, and Apollo Router (multipart-subscriptions/Incremental Delivery over HTTP) ([https://gqlgen.com/recipes/subscriptions/](https://gqlgen.com/recipes/subscriptions/), [https://netflix.github.io/dgs/advanced/subscriptions/](https://netflix.github.io/dgs/advanced/subscriptions/), [https://www.apollographql.com/blog/federated-subscriptions-in-graphos-real-time-data-at-scale](https://www.apollographql.com/blog/federated-subscriptions-in-graphos-real-time-data-at-scale)).

**File uploads.** The community **multipart specification** (jaydenseric/graphql-multipart-request-spec) is widely supported, but Apollo Server 2 had a CVE chain in its bundled `graphql-upload-8-fork` ([https://github.com/apollographql/apollo-server/issues/6485](https://github.com/apollographql/apollo-server/issues/6485)). [GitHub](https://github.com/apollographql/apollo-server/issues/6485)

**Persisted queries and APQ.** APQ uses the SHA-256 of the operation text as a content-addressed key; if the server doesn't have it cached, the client retries with the full query ([https://www.apollographql.com/docs/apollo-server/performance/apq](https://www.apollographql.com/docs/apollo-server/performance/apq)). Note: APQ alone is a *performance* feature; it becomes a *security* feature only when combined with safelisting (the client *must* use a registered hash and full strings are rejected) ([https://www.apollographql.com/docs/graphos/operations/persisted-queries](https://www.apollographql.com/docs/graphos/operations/persisted-queries), [https://the-guild.dev/graphql/yoga-server/docs/features/automatic-persisted-queries](https://the-guild.dev/graphql/yoga-server/docs/features/automatic-persisted-queries)). [Apollo GraphQL + 2](https://www.apollographql.com/docs/apollo-server/performance/apq)

**Introspection.** Querying `__schema` returns the entire SDL. Disable in production for non-public APIs to reduce attack surface ([https://portswigger.net/web-security/graphql](https://portswigger.net/web-security/graphql)). [Apollo GraphQL](https://www.apollographql.com/docs/graphos/resources/saf/security)

## Deep connections to other protocols

**HTTP/1.1.** The default transport. Almost every public GraphQL API exposes a single endpoint at `/graphql` and accepts POST with a JSON body containing `query`, `variables`, `operationName`. GET is allowed for cacheable read-only queries ([https://graphql.org/learn/serving-over-http/](https://graphql.org/learn/serving-over-http/)).

**HTTP/2 (RFC 9113).** Multiplexing means many GraphQL operations can share a TCP connection, but unlike gRPC, GraphQL does not exploit HTTP/2 *streams* for incremental delivery — incremental delivery uses HTTP **chunked transfer-encoding (1.1) or multipart bodies on /2** ([https://github.com/graphql/graphql-over-http/blob/main/rfcs/IncrementalDelivery.md](https://github.com/graphql/graphql-over-http/blob/main/rfcs/IncrementalDelivery.md)). Practical upshot: HTTP/2 helps RPS by removing head-of-line per-connection blocking; it doesn't change the GraphQL execution model.

**JSON-RPC 2.0.** The philosophical sibling: single endpoint, message-based, structured request/response envelope. The contrast is that GraphQL ships a *type system and selection language*, while JSON-RPC defines only the envelope and leaves payload semantics to each method. JSON-RPC has no schema, no introspection, and no field selection.

**WebSocket (RFC 6455).** Upgraded from HTTP, gives a duplex stream. The active GraphQL sub-protocols are `graphql-transport-ws` (the modern `graphql-ws` library, default) and the deprecated `graphql-ws` (the old Apollo `subscriptions-transport-ws` protocol; same name, different protocol — yes, this is genuinely confusing) ([https://wundergraph.com/blog/quirks_of_graphql_subscriptions_sse_websockets_hasura_apollo_federation_supergraph](https://wundergraph.com/blog/quirks_of_graphql_subscriptions_sse_websockets_hasura_apollo_federation_supergraph), [https://github.com/enisdenjo/graphql-ws](https://github.com/enisdenjo/graphql-ws)).

**REST.** GraphQL was *designed* to relieve REST's over- and under-fetching pains for nested mobile UI graphs, especially Facebook's News Feed ([https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/](https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/)). REST is still better when responses can be cached at the HTTP layer trivially, when clients are simple (CRUD), and when public endpoints need predictable, scriptable URLs. Even Lee Byron stresses GraphQL is "not a silver bullet" and that REST excels at simpler public APIs ([https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/](https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/)).

**SSE.** Server-Sent Events are an HTTP-native, one-way streaming primitive that survives most corporate proxies (which often kill WebSocket upgrades). Apollo Router's "multipart-subscriptions" and Spring for GraphQL use SSE-style transports for subscriptions; the GraphQL-over-HTTP working group is moving toward SSE as the recommended transport because it doesn't require a separate stateful connection ([https://www.apollographql.com/blog/federated-subscriptions-in-graphos-real-time-data-at-scale](https://www.apollographql.com/blog/federated-subscriptions-in-graphos-real-time-data-at-scale), [https://netflix.github.io/dgs/advanced/subscriptions/](https://netflix.github.io/dgs/advanced/subscriptions/)). [Apollo GraphQL](https://www.apollographql.com/blog/federated-subscriptions-in-graphos-real-time-data-at-scale)

**SOAP.** A surprising parallel: SOAP was the original "RPC over HTTP with a strong schema (WSDL/XSD)". GraphQL is in some sense SOAP's grandchild — minus the XML, plus a query language and field-selection. Both decouple operation discovery (introspection ↔ WSDL) from operation execution.

**gRPC.** The other modern contender: schema-first via Protocol Buffers, binary, streaming via HTTP/2 streams, RPC-shaped. gRPC excels at internal service-to-service traffic where every byte matters and binary clients are acceptable; GraphQL excels at heterogeneous front-end consumers. Many companies use gRPC inside the data center and GraphQL at the edge.

**tRPC.** A TypeScript-only RPC where the schema *is* the TypeScript code. tRPC is increasingly preferred when frontend and backend are a single codebase — Jens Neuse (WunderGraph) makes this argument explicitly ([https://wundergraph.com/blog/six-year-graphql-recap](https://wundergraph.com/blog/six-year-graphql-recap)). [WunderGraph](https://wundergraph.com/blog/six-year-graphql-recap)

**OpenAPI / Swagger.** Schema-first description for REST. The Composite Schemas / Fusion gateways now allow REST APIs (described by OpenAPI) to be federated alongside GraphQL subgraphs ([https://chillicream.com/blog/2023/08/15/graphql-fusion/](https://chillicream.com/blog/2023/08/15/graphql-fusion/)).

**OData.** Microsoft's earlier (2007) attempt at "queryable REST" with `$select`, `$expand`, `$filter`. Conceptually the closest ancestor of GraphQL, but tied to the entity-data model. GraphQL essentially won the design competition by being mobile-app-shaped instead of enterprise-data-warehouse-shaped.

**Falcor.** Netflix's competitor library released in 2015. JS-only, model-as-a-virtual-JSON-graph. Apollo's own 2016 comparison concluded GraphQL was more powerful and had wider language support; Netflix itself migrated *from* Falcor *to* GraphQL Federation in 2022 ([https://www.apollographql.com/blog/graphql-vs-falcor-4f1e9cbf7504](https://www.apollographql.com/blog/graphql-vs-falcor-4f1e9cbf7504), [https://dev.to/amplication/why-netflix-took-a-bet-on-graphql-56i2](https://dev.to/amplication/why-netflix-took-a-bet-on-graphql-56i2), [https://www.apollographql.com/blog/an-unexpected-journey-how-netflix-transitioned-to-a-federated-supergraph](https://www.apollographql.com/blog/an-unexpected-journey-how-netflix-transitioned-to-a-federated-supergraph)). Falcor "lost". [TechTarget + 4](https://www.techtarget.com/searchapparchitecture/tip/Falcor-vs-GraphQL-The-differences-that-matter)

**JSON:API.** A REST flavor that encodes resource graphs and relationships in JSON. Co-existed peacefully; mostly displaced in greenfield mobile work by GraphQL.

**HTTP/3 + QUIC.** QUIC's 0-RTT and per-stream loss recovery primarily benefit page loads and many small requests — i.e., the REST shape. GraphQL clients tend to send fewer, larger requests, so the benefit is more modest. APQ/persisted queries shrink the request payload enough that GET-with-hash queries are CDN-cacheable on /3 ([https://www.apollographql.com/blog/apollo-router-our-new-high-performance-federation-runtime-is-now-available-in-open-preview](https://www.apollographql.com/blog/apollo-router-our-new-high-performance-federation-runtime-is-now-available-in-open-preview)).

**TLS.** Treat GraphQL like any other HTTP service: TLS 1.3 on :443. There is nothing GraphQL-specific here, except that introspection responses can leak entire schemas, so TLS + auth + introspection-off-in-prod is the production posture ([https://portswigger.net/web-security/graphql](https://portswigger.net/web-security/graphql)).

## Failure modes and famous incidents

**CVE-2023-26144 — graphql-js DoS (the recursive/overlapping fragment bug).** Versions 16.3.0 to <16.8.1 of `graphql` had insufficient checks in `OverlappingFieldsCanBeMergedRule`; deeply-nested or reused fragments triggered exponential validation work ([https://github.com/advisories/GHSA-9pv7-vfvm-6vr7](https://github.com/advisories/GHSA-9pv7-vfvm-6vr7), [https://nvd.nist.gov/vuln/detail/CVE-2023-26144](https://nvd.nist.gov/vuln/detail/CVE-2023-26144), [https://github.com/graphql/graphql-js/issues/3955](https://github.com/graphql/graphql-js/issues/3955)). The graphql-php variant of the same class (with O(n²×m²) cost in nested inline fragments, ~117 sec CPU/request on a 364 KB query at 200/100 nesting) extended the lineage well into 2024 ([https://dailycve.com/graphql-php-algorithmic-complexity-dos-cve-2023-26144-critical/](https://dailycve.com/graphql-php-algorithmic-complexity-dos-cve-2023-26144-critical/)). Network attack vector, no auth, no UI, CVSS 7.5 ([https://www.ameeba.com/blog/cve-2025-31496-graphql-query-vulnerability-in-apollo-compiler-leading-to-possible-denial-of-service/](https://www.ameeba.com/blog/cve-2025-31496-graphql-query-vulnerability-in-apollo-compiler-leading-to-possible-denial-of-service/)). **CVE-2025-32032 — Apollo Router DoS via recursive named fragments.** The native query planner mishandled deeply-nested named fragments; recursive fragments expanded indefinitely during *planning* (before any subgraph fetch), monopolizing the multi-threaded planner pool. Affected versions <1.61.2 and <2.1.1 ([https://undercodenews.com/cve-critical-flaw-in-apollo-router-leaves-federated-graphql-systems-vulnerable/](https://undercodenews.com/cve-critical-flaw-in-apollo-router-leaves-federated-graphql-systems-vulnerable/)). The same class is acknowledged in Federation's `query-planner-js` advisory `GHSA-fmj9-77q8-g6c4` ([https://github.com/apollographql/federation/blob/main/query-planner-js/CHANGELOG.md](https://github.com/apollographql/federation/blob/main/query-planner-js/CHANGELOG.md)). A separate **prototype-pollution** fix landed in Federation as `GHSA-pfjj-6f4p-rvmh` ([https://github.com/apollographql/federation/blob/main/internals-js/CHANGELOG.md](https://github.com/apollographql/federation/blob/main/internals-js/CHANGELOG.md)). [UNDERCODE NEWS + 4](https://undercodenews.com/cve-critical-flaw-in-apollo-router-leaves-federated-graphql-systems-vulnerable/)

**Apollo Server XSS via GraphQL Playground (`GHSA-qm7x-rc44-rrqw`).** A graphql-playground XSS, exploitable via a crafted link, with cookie/session theft impact for any Apollo Server still serving Playground ([https://github.com/apollographql/apollo-server/security/advisories/GHSA-qm7x-rc44-rrqw](https://github.com/apollographql/apollo-server/security/advisories/GHSA-qm7x-rc44-rrqw)). [GitHub](https://github.com/apollographql/apollo-server/security/advisories/GHSA-qm7x-rc44-rrqw)

**`GHSA-47qc-hrx3-r993` (Sept 25, 2025) — CSRF in Apollo Embedded Sandbox / Explorer** via `window.postMessage` origin-validation bypass ([https://github.com/apollographql/apollo-server/security/advisories](https://github.com/apollographql/apollo-server/security/advisories)). [GitHub](https://github.com/apollographql/apollo-server/security/advisories)

**CVE-2022-24434 / `dicer`.** Transitive vulnerability through `apollo-server-core`'s `graphql-upload-8-fork` → `busboy` → `dicer`, hitting many Apollo Server 2 deployments ([https://github.com/apollographql/apollo-server/issues/6485](https://github.com/apollographql/apollo-server/issues/6485)). [GitHub](https://github.com/apollographql/apollo-server/issues/6485)

**Hasura — `update_many` row-level authorization bypass.** Discovered by Morten Hillbom and Issaaf Kattan (Celsia.io / Nhost) in November 2022, affecting Hasura GraphQL Engine **2.10.0–2.15.x**; allowed authenticated users to update any column they had update permission for on *any* row of the target table — and read back any selected columns. Also notable for the *meta-incident*: the documented `security@hasura.io` mailbox was misconfigured and didn't deliver, so the disclosure had to come in via a personal contact ([https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)). A separate older **information-disclosure** issue (Exploit-DB 50803, Hasura Community 2.2.0) leaked arbitrary root environment variables via `add_remote_schema` URL coercion ([https://www.exploit-db.com/exploits/50803](https://www.exploit-db.com/exploits/50803)). [Hasura + 2](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)

**Class: batching attacks / alias overloading.** Single HTTP requests can carry thousands of aliased subqueries — bypassing per-request rate limits, brute-forcing OTP/2FA endpoints, and triggering N+1 amplification at the database layer ([https://escape.tech/blog/graphql-batch-attacks-cause-dos/](https://escape.tech/blog/graphql-batch-attacks-cause-dos/), [https://checkmarx.com/blog/didnt-notice-your-rate-limiting-graphql-batching-attack/](https://checkmarx.com/blog/didnt-notice-your-rate-limiting-graphql-batching-attack/), [https://medium.com/@instatunnel/graphql-batching-attacks-how-100-queries-become-10-000-database-calls-156fb3b2da1c](https://medium.com/@instatunnel/graphql-batching-attacks-how-100-queries-become-10-000-database-calls-156fb3b2da1c), [https://dev.to/ivandotv/preventing-graphql-batching-attacks-56o3](https://dev.to/ivandotv/preventing-graphql-batching-attacks-56o3), [https://www.imperva.com/blog/graphql-vulnerabilities-common-attacks/](https://www.imperva.com/blog/graphql-vulnerabilities-common-attacks/)). [Pentesterlab](https://pentesterlab.com/glossary/graphql-batching-attack)

**Class: introspection / schema leak.** Even with introspection off, **Apollo's "did you mean" suggestions** can be used by tools like Clairvoyance to recover the schema. Apollo has historically had no toggle for this; community workarounds exist ([https://portswigger.net/web-security/graphql](https://portswigger.net/web-security/graphql)). [PortSwigger](https://portswigger.net/web-security/graphql)

**Class: the "GraphQL is insecure by default" debate.** It is frequently true that a default Apollo/Yoga setup ships with introspection on, no depth/complexity limits, no APQ safelisting, alias batching unlimited, and 200 OK on every error — every one of those is a foot-gun ([https://www.apollographql.com/docs/graphos/resources/saf/security](https://www.apollographql.com/docs/graphos/resources/saf/security), [https://medium.com/@xuorig/a-guide-to-graphql-rate-limiting-security-e62a86ef8114](https://medium.com/@xuorig/a-guide-to-graphql-rate-limiting-security-e62a86ef8114)).

**N+1 production incidents.** This is *the* recurring class. The DataLoader pattern (originally `@schrockn`'s Loader at Facebook in 2010) is the textbook fix ([https://github.com/graphql/dataloader](https://github.com/graphql/dataloader)). Specific named outage attribution to N+1 in production at Shopify/GitHub couldn't be pinned to a public post-mortem and is therefore `[needs source]` — but the existence of GitHub's points-cost rate limiter is itself an admission that resolver-amplification at scale is a real production hazard ([https://docs.github.com/en/graphql/overview/rate-limits-and-query-limits-for-the-graphql-api](https://docs.github.com/en/graphql/overview/rate-limits-and-query-limits-for-the-graphql-api)). [GitHub](https://github.com/graphql/dataloader)

## Fun facts and anecdotes

- **The internal name was "SuperGraph"**, not GraphQL. Schrock posted the prototype to Facebook's internal review tool and the team picked the public name later ([https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/](https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/)). Lee Byron's tribute tweet calling it "the prototype that really resembled GraphQL" is the canonical citation. [Postman](https://blog.postman.com/what-is-graphql-part-one-the-facebook-years/)
- **Founders, name-spelled-out:** Nick Schrock (now founder of **Elementl**, the Dagster company), Lee Byron (now Executive Director of the GraphQL Foundation; former engineering lead at Robinhood and Watershed), Dan Schafer (still at Meta) ([https://leebyron.com/](https://leebyron.com/), [https://graphql.org/conf/2023/speakers/lee_byron.25krdom6](https://graphql.org/conf/2023/speakers/lee_byron.25krdom6)). The misspelling **"Dan Schmidt"** appears occasionally (including in the user prompt for this report) — the correct spelling is **Schafer**. [Medium + 2](https://medium.com/@patcito/reacteurope-interview-18-lee-byron-af81838df52a)
- **Lee Byron's "Byronic fantasy."** Schafer recounts in the Honeypot documentary that he had been "trying to persuade Lee to open source this for a while. I think I referred to it as my Byronic fantasy" ([https://www.linkedin.com/learning/graphql-the-documentary/graphql-the-documentary](https://www.linkedin.com/learning/graphql-the-documentary/graphql-the-documentary)).
- **The HTML5 nightmare origin story.** Facebook's iOS/Android apps in 2011 were thin wrappers around a mobile webview, and GraphQL was born from the necessity of replacing them with a native data API ([https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/](https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/), [https://dev.to/visy_ani/the-story-of-graphql-how-facebooks-crisis-changed-apis-forever-2d92](https://dev.to/visy_ani/the-story-of-graphql-how-facebooks-crisis-changed-apis-forever-2d92)). [WP Tavern](https://wptavern.com/lessons-from-the-graphql-documentary-never-underestimate-the-power-of-open-source-communities)[FB](https://engineering.fb.com/2015/09/14/core-infra/graphql-a-data-query-language/)
- **GraphQL is NOT a graph-database query language.** This is the most persistent misunderstanding in the entire ecosystem — confusion between Cypher/Gremlin-style graph databases and GraphQL. The "graph" in GraphQL refers to your object graph, not a backing store; the spec is explicit it's not tied to any database ([https://spec.graphql.org/September2025/](https://spec.graphql.org/September2025/), [https://www.apollographql.com/blog/graphql-vs-falcor-4f1e9cbf7504](https://www.apollographql.com/blog/graphql-vs-falcor-4f1e9cbf7504)). [Apollo GraphQL](https://www.apollographql.com/blog/graphql-vs-falcor-4f1e9cbf7504)[Graphql](https://spec.graphql.org/September2025/)
- **Trailing commas.** GraphQL is one of the few hand-written wire formats that allows trailing commas in argument and field lists, partly to be friendly to codegen.
- **Nullable-by-default.** Every field is nullable unless marked `!` — the spec's authors deliberately chose this so that resolvers could degrade gracefully when a downstream service is down ([https://github.com/graphql/graphql-spec](https://github.com/graphql/graphql-spec)). It remains the most-debated design decision in the community. [GitHub](https://github.com/graphql/graphql-spec)
- **The 5-year Schrock tweet.** "🎁🎉 Happy 5th birthday to @GraphQL! It was five years ago that @schrockn wrote the 'SuperGraph' prototype, and it's been a wild ride since!" ([https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/](https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/)). [Nordic APIs](https://nordicapis.com/interview-with-graphql-co-creator-lee-byron/)

[React](https://legacy.reactjs.org/blog/2015/02/18/react-conf-roundup-2015.html)
- **"Subscription is the hardest part."** This shows up as a running joke in GraphQL Working Group transcripts and at every GraphQLConf — proven by the existence of **at least five** subscription transports in widespread use (graphql-ws, the deprecated subscriptions-transport-ws, SSE-graphql, AppSync MQTT, and Apollo's multipart-subscriptions) ([https://wundergraph.com/blog/quirks_of_graphql_subscriptions_sse_websockets_hasura_apollo_federation_supergraph](https://wundergraph.com/blog/quirks_of_graphql_subscriptions_sse_websockets_hasura_apollo_federation_supergraph)). ## Practical wisdom

What experienced GraphQL engineers actually configure in production:

- **Query depth limiting.** Cap `maxDepth` to 10–15 to defeat the recursive-fragment DoS class ([https://undercodenews.com/cve-critical-flaw-in-apollo-router-leaves-federated-graphql-systems-vulnerable/](https://undercodenews.com/cve-critical-flaw-in-apollo-router-leaves-federated-graphql-systems-vulnerable/), [https://medium.com/@xuorig/a-guide-to-graphql-rate-limiting-security-e62a86ef8114](https://medium.com/@xuorig/a-guide-to-graphql-rate-limiting-security-e62a86ef8114)).
- **Cost / complexity analysis.** GitHub's points model and Shopify's calculated query cost are the public archetypes ([https://docs.github.com/en/graphql/overview/rate-limits-and-query-limits-for-the-graphql-api](https://docs.github.com/en/graphql/overview/rate-limits-and-query-limits-for-the-graphql-api), [https://shopify.engineering/rate-limiting-graphql-apis-calculating-query-complexity](https://shopify.engineering/rate-limiting-graphql-apis-calculating-query-complexity)). Federation v2.9+ now ships `@cost` and `@listSize` directives so subgraph authors can declare cost in the schema ([https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions](https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions)). [Apollo GraphQL](https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions)
- **Persisted queries in production.** Use **safelisted persisted queries** (not just APQ) on internal apps to convert your GraphQL endpoint into something with the cacheability and security profile of a REST endpoint ([https://www.apollographql.com/docs/graphos/operations/persisted-queries](https://www.apollographql.com/docs/graphos/operations/persisted-queries)).
- **APQ as a security feature.** APQ alone is not security; safelisted persisted queries are. Apollo's docs are explicit ([https://the-guild.dev/graphql/yoga-server/docs/features/automatic-persisted-queries](https://the-guild.dev/graphql/yoga-server/docs/features/automatic-persisted-queries)).
- **N+1 → DataLoader.** Wrap every batched fetch in a DataLoader instance scoped to the request; Schrock's original 2010 pattern is still state of the art, although WunderGraph's "DataLoader 3.0" breadth-first algorithm in Cosmo Router is a 2023+ refinement ([https://github.com/graphql/dataloader](https://github.com/graphql/dataloader), [https://wundergraph.com/blog/dataloader_3_0_breadth_first_data_loading](https://wundergraph.com/blog/dataloader_3_0_breadth_first_data_loading)).
- **Federated subgraph contract testing.** Schema composition in CI (Rover/Cosmo/Hive) catches breaking changes pre-merge ([https://www.apollographql.com/blog/apollo-federation-2-is-now-generally-available](https://www.apollographql.com/blog/apollo-federation-2-is-now-generally-available)).
- **Per-field tracing.** Apollo trace format and the OpenTelemetry GraphQL semantic conventions (`graphql.document`, `graphql.operation.name`, `graphql.operation.type`) are now the standard knobs ([https://opentelemetry.io/docs/specs/semconv/graphql/](https://opentelemetry.io/docs/specs/semconv/graphql/), [https://opentelemetry.io/docs/specs/semconv/registry/attributes/graphql/](https://opentelemetry.io/docs/specs/semconv/registry/attributes/graphql/), [https://github.com/graphql/otel-wg](https://github.com/graphql/otel-wg)). Apollo specifically warns against making `graphql.document` a default span attribute due to high cardinality and PII risk ([https://www.apollographql.com/docs/graphos/routing/observability/router-telemetry-otel/enabling-telemetry/standard-attributes](https://www.apollographql.com/docs/graphos/routing/observability/router-telemetry-otel/enabling-telemetry/standard-attributes)). [Apollo GraphQL](https://www.apollographql.com/docs/graphos/routing/observability/router-telemetry-otel/enabling-telemetry/standard-attributes)
- **Pagination — Relay-style cursor connections.** `edges/node/cursor/pageInfo` is the de-facto standard, even if you're not using Relay ([https://relay.dev/graphql/connections.htm](https://relay.dev/graphql/connections.htm), [https://graphql.org/learn/pagination/](https://graphql.org/learn/pagination/)). Michael Staib's "relative cursors" (GraphQLConf 2025) keep jump-to-page ergonomics on top of cursors ([https://graphql.org/blog/2025-10-20-graphql-conf-2025-article-1/](https://graphql.org/blog/2025-10-20-graphql-conf-2025-article-1/)). [GraphQL](https://graphql.org/blog/2025-10-20-graphql-conf-2025-article-1/)
- **Errors[] vs throwing.** Use field errors for partial success; use top-level errors for validation/auth. Most production teams add a *typed-result* pattern (a "UserResult" union of "User" and "UserError") for business errors, separate from infrastructure errors.
- **Nullable-by-default debate.** Real teams settle this with a linter rule that *requires* a justification comment whenever a field is non-null on an object that crosses a service boundary.
- **Don't version, evolve.** Use `@deprecated(reason: "...")` and add fields rather than versioning ([https://en.wikipedia.org/wiki/GraphQL](https://en.wikipedia.org/wiki/GraphQL)). [Wikipedia](https://en.wikipedia.org/wiki/GraphQL)

## Where things are heading (2025–2026 frontier)

**1. Composite Schemas Working Group is the center of gravity.** Reconvened May 2024, with formal participation from Apollo, ChilliCream, Google, Graphile, The Guild, Hasura, IBM. The work-in-progress spec lives at [https://github.com/graphql/composite-schemas-spec](https://github.com/graphql/composite-schemas-spec); meetings are public ([https://graphql.org/blog/2024-05-16-composite-schemas-announcement/](https://graphql.org/blog/2024-05-16-composite-schemas-announcement/), [https://github.com/graphql/composite-schemas-wg](https://github.com/graphql/composite-schemas-wg)). The CSS introduces *Entity Resolvers* and the `@is` directive — entities become "regular resolvers" with semantic-equivalence annotations, dropping the magic of `_entities` ([https://wundergraph.com/blog/the_state_of_distributed_graphql_2024](https://wundergraph.com/blog/the_state_of_distributed_graphql_2024)). [GraphQL + 3](https://graphql.org/blog/2024-05-16-composite-schemas-announcement/)

**2. Apollo Federation v2.x — current state.** Federation versioning runs through directive `@link` URLs. Latest 2.x bumps include `@cost` and `@listSize` (v2.9), progressive `@override`, `@authenticated`, `@requiresScopes`, `@policy`, and Connect spec versions (v2.10–2.13) ([https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions](https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions)). **Apollo Router 1.60+ no longer runs Federation v1 supergraphs** — a significant deprecation ([https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions](https://www.apollographql.com/docs/graphos/schema-design/federated-schemas/reference/versions)). The original `@apollo/federation` package was end-of-lifed in September 2023 ([https://www.npmjs.com/package/@apollo/federation](https://www.npmjs.com/package/@apollo/federation)). [GitHub + 2](https://github.com/graphql/graphql-spec/pull/1110)

**60-second narrated hook (write for the ear):**

> *In February 2012, three engineers at Facebook — Nick Schrock, Lee Byron, and Dan Schafer — got assigned what looked like a boring problem: rebuild the iOS News Feed API. Schrock built a prototype in a few days and called it… SuperGraph. Three years later, in January 2015 at React.js Conf, Facebook finally unwrapped it for the world and gave it a duller name: GraphQL. Ten years on, GraphQL powers GitHub's API, Shopify's storefront, Netflix's content engine, every major Apollo customer's supergraph — about half of the Fortune 500's API tier. **Pause-and-think moment:**

> GraphQL returns HTTP 200 even when your query failed. The full GraphQL-over-HTTP spec actually *requires* it: if `data` is non-null, the server **must** use a 200 status code ([https://github.com/graphql/graphql-over-http/blob/main/spec/GraphQLOverHTTP.md](https://github.com/graphql/graphql-over-http/blob/main/spec/GraphQLOverHTTP.md)). That's why your APM dashboard shows a green graph while users are seeing error toasts. The spec only fixes this if you opt into the new `application/graphql-response+json` media type — and that became the recommended default on **January 1, 2025**. [Progosling](https://progosling.com/en/dev-digest/graphql-over-http-working-draft-2025)[GitHub](https://github.com/graphql/graphql-over-http/blob/main/spec/GraphQLOverHTTP.md)

**Failure-story arc (Hasura, November 2022):**

- **Setup.** Hasura ships v2.10 with a new "update_many" API for Postgres backends. It's a great feature: bulk updates against any rows the user is permitted to mutate. [Hasura](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)
- **Mistake.** A bug in the row-level authorization code meant the permission check silently slipped: any authenticated user could update any column they had update permission on, on **any** row of the target table — and read back any selectable column from the affected rows ([https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)). [Hasura](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)
- **Consequence.** It went unnoticed for over a year. When researchers Morten Hillbom and Issaaf Kattan from Nhost's customer Celsia.io discovered it on November 16, 2022, they emailed `security@hasura.io`. The mailbox was misconfigured. The email vanished. They had to track down a Hasura employee through a personal contact at Nhost just to *report* the vulnerability. [Hasura](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)
- **Resolution.** Hasura patched 2.10.2 through 2.15.2 within days of being told, pulled vulnerable Docker images, redesigned authorization to enforce coverage in CI, and — most importantly — replaced their `security@` mailbox with a real human-monitored alias. The post-mortem is unusual for being open about both failures: the *code* failure and the *process* failure that hid it. ([https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)) [Hasura](https://hasura.io/blog/critical-vulnerability-in-hasuras-graphql-engine-v2-10-0)

## Caveats

- **Vendor benchmarks are vendor benchmarks.** Apollo Router's "9×/10×" numbers, Cosmo's "2×P99 better than Apollo" claims, and Grafbase's gateway results are each from the publishing vendor; treat them as directional, not authoritative. Independent comparisons exist ([https://grafbase.com/blog/benchmarking-grafbase-vs-apollo-vs-cosmo-vs-mesh](https://grafbase.com/blog/benchmarking-grafbase-vs-apollo-vs-cosmo-vs-mesh)), but no fully neutral, reproducible Federation benchmark has consensus as of May 2026.
- **"Coinbase deprecated GraphQL"** and similar specific company-retreat stories should be verified against primary engineering-blog posts. The general "honeymoon is over" trend is real and supported; specific obituaries usually are not.
- **Spec status moves quickly.** The September 2025 spec edition is published, but the GraphQL-over-HTTP spec, Composite Schemas spec, and `@defer`/`@stream` are *all still drafts* as of the May 2026 cut. Verify versions before quoting.
- **The "Dan Schmidt" name in the briefing was wrong**; the correct co-creator name is **Dan Schafer** ([https://medium.com/@leeb/introducing-the-graphql-foundation-3235d8186d6d](https://medium.com/@leeb/introducing-the-graphql-foundation-3235d8186d6d)). Several derivative blogs perpetuate the error.
- **The original BSD+Patents license relicensing happened on September 26, 2017**, not 2018; the 2018/2019 milestones are the *Foundation* moves, which are distinct ([https://leebyron.com/relicensing-the-graphql-specification/](https://leebyron.com/relicensing-the-graphql-specification/), [https://www.linuxfoundation.org/press/press-release/intent_to_form_graphql](https://www.linuxfoundation.org/press/press-release/intent_to_form_graphql)).
- **Production-deployment numbers** are sparse outside vendor case studies; the GitHub rate-limit numbers and Shopify's published cost formula are the rare publicly-documented hard numbers ([https://docs.github.com/en/graphql/overview/rate-limits-and-query-limits-for-the-graphql-api](https://docs.github.com/en/graphql/overview/rate-limits-and-query-limits-for-the-graphql-api), [https://shopify.engineering/rate-limiting-graphql-apis-calculating-query-complexity](https://shopify.engineering/rate-limiting-graphql-apis-calculating-query-complexity)). Specific incident attributions (e.g. "GitHub had an N+1 outage on date X") were not found in public post-mortems and are marked `[needs source]` rather than fabricated.
- **The exact attendance at GraphQLConf 2025** is given as "over 250" by the GraphQL Foundation's own recap ([https://graphql.org/blog/2025-10-20-graphql-conf-2025-article-1/](https://graphql.org/blog/2025-10-20-graphql-conf-2025-article-1/)); attendees suggest the number was higher across the three days but precise figures aren't published.
- **GraphQL Radio podcast's current status** is unverified; treat the "is GraphQL Radio still going?" question as `[needs source]` for the podcast itself.
